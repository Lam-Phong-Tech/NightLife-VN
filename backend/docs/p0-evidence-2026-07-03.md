## Runtime Cases To Capture Manually

These cases are covered in code/tests, and can be captured with Postman or a
browser against a running environment:

- Bill without booking but with `couponIssueId`: `POST /member/bills` accepts
  `couponId` + `couponIssueId`, creates a submitted bill with `bookingId` empty,
  and keeps the coupon/coupon issue relation.
- Admin bill review relation visibility: `GET /admin/sensitive-bills` returns
  `booking`, `coupon`, and `couponIssue` relation objects when present. The
  Admin UI review table now shows the relation data and displays the
  `Khong co booking` badge directly on rows that have no booking.

## Relevant Test Anchors

- Backend: `backend/src/nightlife-data/nightlife-data.service.spec.ts`
  - `submits a member bill with a coupon issue without requiring a booking`
  - `listSensitiveBillsForAdmin` masks sensitive data while returning relation
